# Migration Guide: Nyx â†’ Malix

This guide provides a component-by-component mapping from the legacy **Nyx** design system to **Malix**, along with migration notes and what to watch for.

## Before You Start

1. **Install Malix** alongside Nyx â€” they can coexist during migration
2. **Import Malix CSS** in your entry point: `import '@camtomlabs/malix-design-system/css'`
3. **Migrate page-by-page**, not all at once â€” both systems can render on the same page
4. **Use the Malix Stylelint config** to catch hardcoded values: `extends: ['@camtomlabs/malix-design-system/stylelint.config']`

```bash
pnpm add @camtomlabs/malix-design-system
```

## Component Mapping

### Buttons

| Nyx Component | Malix Equivalent | Notes |
|---|---|---|
| `<NyxButton>` | `<Button>` | Use `hierarchy` prop instead of `variant` for visual style |
| `<NyxIconButton>` | `<Button variant="icon-only">` | Pass `icon` and `label` props |
| `<NyxButtonGroup>` | Flex container + `<Button>` | No dedicated wrapper needed â€” use flex/gap |

**Key differences:**
- Nyx uses `variant="primary"` â†’ Malix uses `hierarchy="primary"`
- Nyx `size="small"` â†’ Malix `size="sm"`
- Nyx `isLoading` â†’ Malix `loading`

```tsx
// Nyx
<NyxButton variant="primary" size="small" isLoading>Save</NyxButton>

// Malix
<Button hierarchy="primary" size="sm" loading>Save</Button>
```

## Not Yet in Malix

The following Nyx components do not have Malix equivalents yet. Continue using Nyx for these until a Malix replacement is available:

| Nyx Component | Status | Workaround |
|---|---|---|
| `<NyxToast>` | Planned | Use `<Banner>` positioned fixed, or a third-party toast library |
| `<NyxDrawer>` | Planned | Use `<SidebarPanel>` with absolute positioning |
| `<NyxCalendar>` | Under review | Use `<DateInput>` for simple date selection |
| `<NyxColorPicker>` | Not planned | Use a third-party color picker |
| `<NyxRichTextEditor>` | Not planned | Use a third-party editor |

## CSS / Token Migration

### Replace Nyx CSS variables

| Nyx Token | Malix Token |
|---|---|
| `--nyx-bg` | `--malix-background-main` |
| `--nyx-surface` | `--malix-surface` |
| `--nyx-text` | `--malix-foreground` |
| `--nyx-text-secondary` | `--malix-foreground-secondary` |
| `--nyx-border` | `--malix-border` |
| `--nyx-primary` | `--malix-primary` |
| `--nyx-error` | `--malix-error` |
| `--nyx-success` | `--malix-success` |
| `--nyx-warning` | `--malix-warning` |

### Replace hardcoded values

```css
/* Before (Nyx / hardcoded) */
.my-component {
  color: #111827;
  background: #ffffff;
  border: 1px solid #e5e7eb;
  z-index: 9999;
}

/* After (Malix tokens) */
.my-component {
  color: var(--malix-foreground);
  background: var(--malix-surface);
  border: 1px solid var(--malix-border);
  z-index: var(--malix-z-modal);
}
```

### Remove Bootstrap overrides

If your project uses Bootstrap alongside Nyx, Malix provides all the primitives you need. You can progressively remove Bootstrap:

1. Replace Bootstrap grid with CSS Grid or Flexbox
2. Replace Bootstrap utilities with Malix Tailwind preset utilities
3. Remove `bootstrap.css` import once all pages are migrated
